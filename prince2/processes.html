PRINCE2 Processes

PRINCE2 defines seven processes that describe the step-by-step progression through the project lifecycle from start-up to closure. Each process integrates the seven practices, creating and updating management products at defined points.

The Seven Processes

Starting Up a Project (SU)

Aspect Details
Purpose Ensure prerequisites for initiating the project are in place
Triggered By Project mandate from corporate/programme management
Performed By Executive and Project Manager
Key Activities • Appoint Executive and Project Manager
• Capture previous lessons
• Design and appoint project management team
• Prepare outline Business Case
• Select project approach
• Plan initiation stage
• Assemble Project Brief
Key Products Created Daily Log, Lessons Register, Project Brief (with outline Business Case), Project Product Description, Initiation Stage Plan
Decision Point Project Board: Authorize initiation or reject project

Directing a Project (DP)

Aspect Details
Purpose Enable Project Board to exercise overall control while delegating day-to-day management
Duration Runs from project start to close (parallel to other processes)
Performed By Project Board (Executive, Senior User, Senior Supplier)
Key Activities • Authorize initiation
• Authorize the project (approve PID)
• Authorize stage or exception plan
• Give ad-hoc direction
• Authorize project closure
Management by Exception Board intervenes only at stage boundaries, exceptions, or when ad-hoc direction requested
Key Inputs PID, Stage Plans, End Stage Reports, Exception Reports, Highlight Reports

Initiating a Project (IP)

Aspect Details
Purpose Establish solid foundations for the project with comprehensive planning and documentation
Performed By Project Manager (with team support)
Key Activities • Agree the management approaches
• Establish the project controls
• Create the full Business Case
• Prepare the Benefits Management Approach
• Prepare the Project Plan
• Prepare the first Stage Plan
• Assemble the Project Initiation Documentation (PID)
• Create all registers (Risk, Issue, Quality, etc.)
Key Products Created PID, Business Case, all Management Approaches, all Registers, Project Plan, first delivery Stage Plan, Product Descriptions
Practice Integration All seven practices are applied: Business Case, Organizing, Plans, Quality, Risk, Issues, Progress
Decision Point Project Board: Authorize the project or stop

Controlling a Stage (CS)

Aspect Details
Purpose Assign work, monitor progress, deal with issues, report, and take corrective action
Duration Throughout each management stage
Performed By Project Manager
Key Activities • Authorize Work Packages
• Review Work Package status
• Receive completed Work Packages
• Capture and examine issues and risks
• Review stage status
• Report Highlights to Project Board
• Take corrective action
• Escalate issues and risks
• Raise exception if tolerance breached
Key Products Updated Risk Register, Issue Register, Quality Register, Daily Log, Work Packages
Key Products Created Highlight Reports, Exception Reports (if needed), Issue Reports

Managing Product Delivery (MP)

Aspect Details
Purpose Control the link between Project Manager and Team Manager(s), ensuring planned products are delivered
Performed By Team Manager (or Project Manager if no Team Manager role)
Key Activities • Accept a Work Package
• Execute the Work Package
• Deliver the completed Work Package
• Create Team Plans (optional)
• Report progress via Checkpoint Reports
• Conduct quality activities
• Escalate issues beyond Work Package tolerance
Key Products Checkpoint Reports, Team Plans (optional), completed products
Relationship with CS MP runs in parallel with CS; Team Manager reports to Project Manager

Managing a Stage Boundary (SB)

Aspect Details
Purpose Provide Project Board with key decision points to approve next stage or close project
Performed By Project Manager
Triggered By Approaching end of current stage or exception situation
Key Activities • Plan the next management stage
• Update the Project Plan
• Update the Business Case
• Report stage end (End Stage Report)
• Produce Exception Plan (if exception occurred)
• Review and update project approach
• Capture lessons
Key Products Created End Stage Report, next Stage Plan, updated Project Plan, updated Business Case, Exception Plan (if applicable)
Decision Point Project Board: Authorize next stage, request replanning, or close project

Closing a Project (CP)

Aspect Details
Purpose Confirm project objectives achieved, hand over products, and provide assessment of project performance
Performed By Project Manager
Triggered By End of final delivery stage or premature close decision
Key Activities • Hand over products to operations
• Evaluate the project
• Recommend project closure
• Prepare planned closure (or premature closure)
• Update Benefits Review Plan
• Capture lessons
• Prepare End Project Report
Key Products Created End Project Report, Lessons Report, updated Benefits Review Plan
Types of Closure Planned closure (objectives achieved) or Premature closure (project stopped early)
Decision Point Project Board: Authorize project closure and release resources

Process Flow

Sequential Flow

  1. Starting Up a Project (SU) → Creates Project Brief
  2. Directing a Project (DP) → Authorizes initiation
  3. Initiating a Project (IP) → Creates PID and all foundations
  4. DP → Authorizes the project
  5. Controlling a Stage (CS) + Managing Product Delivery (MP) → Deliver products
  6. Managing a Stage Boundary (SB) → Review and plan next stage
  7. DP → Authorizes next stage
  8. Repeat CS + MP + SB + DP for each management stage
  9. Closing a Project (CP) → Formal closure and handover
  10. DP → Authorizes project closure

Parallel Processes

  • Directing a Project (DP) runs from start to finish, providing oversight
  • Controlling a Stage (CS) and Managing Product Delivery (MP) run in parallel during each stage
  • Managing a Stage Boundary (SB) typically overlaps with end of current CS and before next CS

Key Decision Points

Decision Point Process Decision Maker Options
Authorize Initiation DP (after SU) Project Board Proceed with initiation / Stop project
Authorize Project DP (after IP) Project Board Approve project / Request changes / Stop
Authorize Stage DP (after SB) Project Board Approve next stage / Request replanning / Close project
Authorize Exception Plan DP (after exception) Project Board (stage) or Corporate (project) Approve exception plan / Request alternatives / Close project
Authorize Closure DP (after CP) Project Board Approve closure / Request additional work
