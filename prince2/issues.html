Issues Practice

The Issues practice provides a systematic approach for capturing, examining, and resolving any relevant event that occurs during the project. PRINCE2 7th Edition defines five distinct types of issues that need formal tracking and resolution.

Five Types of Issues (PRINCE2 7th Edition)

PRINCE2 7th Edition expanded the issue classification from three to five types to better reflect modern project contexts:

1. Request for Change (RFC)

Aspect Details
Definition A proposal to modify a baselined product, plan, or other approved project element
Examples • User requests additional system functionality
• Request to modify approved design specifications
• Change to project scope after baseline approval
• Alteration to agreed acceptance criteria
Typical Source Stakeholders, users, suppliers, project team
Impact May affect scope, cost, schedule, quality, or benefits
Decision Authority Project Manager (minor), Change Authority (moderate), Project Board (major)

2. Off-Specification

Aspect Details
Definition Something that should be provided by the project but is missing, or is forecast to be delivered not meeting its specification
Examples • Required report cannot be produced due to software limitations
• Product fails quality testing against specification
• Expected functionality not deliverable within constraints
• Product delivered but missing agreed features
Typical Source Quality reviews, testing, team discovery during delivery
Impact Affects quality, may require scope adjustment or acceptance of reduced specification
Decision Authority Usually requires Project Board approval as it affects agreed baselines

3. Problem or Concern

Aspect Details
Definition Any other issue the Project Manager needs to resolve or escalate
Problem Issue with immediate negative impact requiring resolution
Concern Potential issue whose impact or timing needs evaluation
Examples • Key resource suddenly unavailable (Problem)
• Supplier showing signs of potential delay (Concern)
• Budget overrun in progress (Problem)
• Stakeholder expressing dissatisfaction (Concern)
• Technical difficulty identified (Problem)
• Market conditions changing unfavorably (Concern)
Typical Source Anyone on project team, stakeholders, suppliers, Project Manager observations
Decision Authority Varies by severity and impact - Project Manager to Project Board

4. Business Opportunity (NEW in 7th Edition)

Aspect Details
Definition An unplanned event that presents a potential benefit to the project or business
Examples • New technology becomes available reducing costs
• Opportunity to deliver additional benefits at low cost
• Unexpected resource availability enabling acceleration
• Market shift creating additional value opportunity
• Synergy discovered with another initiative
• Cost savings identified through process improvement
Typical Source Team innovation, market intelligence, stakeholder input, supplier proposals
Impact Positive - potential to enhance benefits, reduce costs, or improve outcomes
Decision Authority Project Manager (minor), Project Board (if affecting business case or strategy)
Why Added Recognizes that not all deviations are negative; encourages proactive value maximization

5. External Event (NEW in 7th Edition)

Aspect Details
Definition An unforeseen development outside the project's control that may influence project delivery
Examples • Regulatory change affecting project requirements
• New government policy impacting scope or approach
• Economic downturn affecting funding
• Natural disaster affecting delivery location
• Competitor action changing market dynamics
• Organizational restructuring affecting stakeholders
• Global supply chain disruption
Typical Source External environment, regulatory bodies, organizational changes, market forces
Impact Can be positive or negative; often requires business case review
Decision Authority Usually Project Board; may escalate to corporate/programme if affecting viability
Why Added Reflects modern project environments with increasing external volatility and uncertainty

Issue Type Summary

Issue Type Nature Control Focus Version
Request for Change Proposed baseline modification Change control and impact assessment All versions
Off-Specification Failure to meet specification Quality management and acceptance All versions
Problem/Concern General negative issue Resolution and escalation All versions

Issue and Change Control Procedure (5 Steps)

Step 1: Capture

Aspect Details
When Continuously throughout project - any time a deviation or query arises
Who Anyone can raise; Project Manager (or delegate) captures formally in Issue Register
Activities • Record issue details in Issue Register
• Assign unique issue identifier
• Document who raised the issue and when
• Provide initial description
• Classify issue type (RFC, Off-Spec, Problem/Concern, Opportunity, External Event)
• Set initial priority
Output Issue registered in Issue Register with unique ID and type classification

Step 2: Examine

Aspect Details
When Immediately after capture; part of CS process
Who Project Manager leads; may involve specialists, Team Managers, or Project Assurance
Activities • Investigate root cause
• Assess impact on seven performance variables (time, cost, quality, scope, benefits, risk, sustainability)
• Determine urgency and priority
• Identify affected products and work packages
• Estimate effort required to resolve
• Check if within change budget/authority
• Document findings in Issue Report (if needed)
Impact Analysis Effect on Business Case, plans, risks, quality, benefits, and other projects/programmes
Output Issue Report with detailed analysis and impact assessment

Step 3: Propose

Aspect Details
When After examination is complete
Who Project Manager develops options; may consult with Change Authority or specialists
Activities • Develop alternative solutions
• Evaluate pros/cons of each option
• Cost and schedule each option
• Assess risks of each option
• Include "do nothing" option
• For Business Opportunities: assess benefit potential and implementation cost
• For External Events: identify adaptation strategies
• Recommend preferred solution
• Document in Issue Report
Options Typically Include • Implement the change/fix
• Implement with modifications
• Defer to later stage/phase
• Reject (do nothing)
• Exploit opportunity (for Business Opportunities)
• Escalate to Project Board or higher authority
Output Issue Report with recommended action and rationale

Step 4: Decide

Aspect Details
When After proposal is prepared
Who Depends on authority levels: Project Manager, Change Authority, or Project Board
Decision Authority • Project Manager: Minor issues within delegated limits
• Change Authority: Issues within defined budget/scope thresholds
• Project Board: Major issues affecting baselines or tolerance
• Corporate/Programme: Issues affecting business case viability or strategic alignment
Activities • Review impact assessment and options
• Consider alignment with business objectives
• Evaluate cost/benefit
• Make formal decision
• Document decision and rationale
• Update Issue Register
Output Approved decision recorded in Issue Register; updated Issue Report

Step 5: Implement

Aspect Details
When After decision is approved; during CS and MP
Who Assigned actionee executes; Project Manager monitors; Team Manager coordinates team work
Activities • Execute approved resolution
• Update affected plans and baselines
• Communicate changes to affected parties
• Update configuration management records
• Track progress of implementation
• Verify resolution is effective
• Close issue when complete
• Update Issue Register with closure information
If Change Approved Update Product Descriptions, plans, Business Case, risk register, and relevant baselines
Output Issue closed in Issue Register; updated project documentation; lessons captured if applicable

Configuration Management

Purpose and Activities

Activity Description
Planning Define what will be controlled, naming conventions, storage locations, access controls, version numbering
Identification Assign unique identifiers to configuration items; establish baselines
Control Manage changes to baselined items; prevent unauthorized modifications; maintain version history
Status Accounting Track current status of items; record changes; maintain audit trail
Verification & Audit Confirm configuration records match actual items; ensure completeness and accuracy

Configuration Item Records

Aspect Details
Owner Project Manager (or Configuration Librarian if appointed)
Created Initiating a Project (IP) for each configuration item
Updated When items are created, modified, approved, or baselined
Contains • Configuration item identifier
• Item type and description
• Version number
• Status (draft, approved, baselined)
• Location/storage
• Relationships to other items
• Change history
• Date and owner information

Change Management Approach

Purpose and Contents

Section Description
Issue & Change Procedure Tailored 5-step process for this project with specific guidance
Issue Types & Scales Definition of all five issue categories; priority and severity scales
Authority Levels Who can approve what: Project Manager limits, Change Authority thresholds, Board escalation triggers
Change Budget Allocated funds for approved changes; thresholds for different authority levels
Roles & Responsibilities Who captures, examines, proposes, decides, implements; Change Authority membership
Tools & Techniques Issue tracking software, impact analysis templates, decision matrices
Configuration Management How items are controlled, versioned, and baselined; CM tools and procedures
Reporting How issues are reported in Highlight Reports, End Stage Reports; escalation procedures

Ownership and Lifecycle

Aspect Details
Owner Project Manager
Created Initiating a Project (IP)
Updated Only if change control approach fundamentally changes
Approved By Project Board as part of Project Initiation Documentation

Related Management Products

Issue Register

Aspect Details
Owner Project Manager
Created Initiating a Project (IP)
Updated Continuously throughout CS as issues are raised, examined, decided, and implemented
Contains • Issue identifier
• Issue type (RFC, Off-Spec, Problem/Concern, Opportunity, External Event)
• Issue description
• Date raised and by whom
• Priority and severity
• Status (open, under review, decided, closed)
• Impact assessment summary
• Decision and decision authority
• Assigned actionee
• Target and actual closure dates

Issue Report

Aspect Details
Owner Project Manager (creator)
Created During CS when detailed analysis is needed (not all issues require a report)
Purpose Provides detailed analysis, impact assessment, options, and recommendations for significant issues
Contains • Issue details from Issue Register
• Detailed impact analysis (seven performance variables)
• Root cause analysis
• Options with pros/cons
• Cost and schedule implications
• Recommendation
• Decision record

Integration with Other Practices

Practice Integration Point
Business Case Major issues (especially External Events) may require Business Case revision; change impacts assessed against justification
Plans Approved changes update plans; off-specifications may require replanning; Business Opportunities may accelerate delivery
Quality Off-specifications are quality failures; Product Descriptions updated via change control
Risk Materialized risks become Problems/Concerns; External Events create new risks; issues may create new risks
Progress Issue status reported in progress reports; significant issues may trigger exceptions
Organizing External Events may require stakeholder re-engagement; Business Opportunities may involve new stakeholders
