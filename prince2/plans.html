Plans Practice

The Plans practice provides a framework for designing and maintaining project, stage, and team plans. PRINCE2 uses Product-Based Planning as its core planning technique, focusing on defining products (deliverables) before activities.

Product-Based Planning (PBP) Technique

Product-Based Planning is PRINCE2's formal planning technique consisting of four sequential steps:

Step 1: Write Project Product Description (PPD)

Aspect Details
When Starting Up a Project (SU) - first planning activity
Who Project Manager with input from Senior User and Senior Supplier
Activities • Define the final product/outcome of the project
• Document customer quality expectations
• Specify acceptance criteria
• Identify acceptance method
• Define acceptance responsibilities
Output Project Product Description - high-level definition of what the project will deliver

Step 2: Create Product Breakdown Structure (PBS)

Aspect Details
When Initiating a Project (IP) or when creating Stage Plans (SB)
Who Project Manager with Team Managers and technical specialists
Activities • Decompose the project product into components
• Identify all specialist products needed
• Identify management products (reports, logs, etc.)
• Create hierarchical structure
• Ensure nothing is missing
• Group related products
Output Product Breakdown Structure - hierarchical diagram showing all products

Step 3: Write Product Descriptions (PDs)

Aspect Details
When Initiating a Project (IP) for major products; detailed during stage planning (SB)
Who Project Manager with input from users (quality expectations) and suppliers (technical specifications)
Activities • Document purpose of each product
• Define composition (what it contains)
• Specify derivation (where it comes from)
• Define quality criteria
• Set quality tolerance
• Identify quality method
• Assign quality responsibilities
Output Product Descriptions for all products - detailed specifications with quality criteria

Step 4: Create Product Flow Diagram (PFD)

Aspect Details
When After Product Descriptions are drafted during IP or SB
Who Project Manager with Team Managers
Activities • Identify dependencies between products
• Determine sequence of product creation
• Show which products feed into others
• Identify external dependencies
• Highlight parallel work opportunities
• Validate logical flow
Output Product Flow Diagram - shows sequence and dependencies of product delivery

Estimation Techniques

Expert Judgment

Aspect Description
Purpose Use experience and knowledge of subject matter experts to estimate effort, duration, and cost
When Used For unique products, innovative work, or when historical data is unavailable
Process Consult with experienced team members, specialists, or external consultants
Advantages Quick; leverages organizational knowledge; good for complex or novel work
Limitations Subject to bias; varies by individual; difficult to verify accuracy

Analogous Estimating

Aspect Description
Purpose Base estimates on historical data from similar past projects
When Used Early project stages when detailed information is limited
Process Review completed projects; identify similarities; adjust for differences; apply ratios or factors
Advantages Fast; uses proven data; good for high-level estimates
Limitations Accuracy depends on similarity; may miss unique project characteristics

Parametric Estimating

Aspect Description
Purpose Use statistical relationships and unit costs to calculate estimates
When Used For repeatable work with established metrics (e.g., cost per square meter, hours per function point)
Process Identify relevant parameters; apply historical unit costs; calculate total estimate
Advantages Objective; scalable; accurate when good data exists
Limitations Requires reliable historical data; may not account for project-specific factors

Bottom-Up Estimating

Aspect Description
Purpose Estimate individual work packages or activities and aggregate upwards
When Used Stage planning when detailed product descriptions are available
Process Break down work into smallest components; estimate each; sum to get total
Advantages Most accurate; detailed understanding of work; team buy-in
Limitations Time-consuming; requires detailed scope definition; can miss high-level factors

Scheduling Techniques

Critical Path Method (CPM)

Aspect Description
Purpose Identify the longest sequence of dependent activities determining minimum project duration
Process • List all activities from Product Flow Diagram
• Determine dependencies
• Estimate duration for each activity
• Calculate earliest start/finish dates
• Calculate latest start/finish dates
• Identify activities with zero float (critical path)
Benefits Shows which activities drive project completion; identifies scheduling flexibility; enables "what-if" analysis
Used For Project Plans and Stage Plans to identify schedule risks and optimize timelines

Critical Chain Method

Aspect Description
Purpose Extension of CPM that accounts for resource constraints and includes buffers
Key Concepts • Resource-constrained scheduling
• Removal of safety time from individual tasks
• Project buffer at end of critical chain
• Feeding buffers where non-critical chains join critical chain
Benefits More realistic schedules; better resource utilization; focused buffer management
Used For Projects with tight resource constraints or aggressive timelines

Types of Plans in PRINCE2

Project Plan

Aspect Details
Owner Project Manager
Created Initiating a Project (IP)
Updated End of each stage (SB) to reflect remaining work and revised forecasts
Approved By Project Board
Purpose High-level plan showing major products, stage boundaries, milestones, overall costs and timescales
Level of Detail Strategic - management stage level, not detailed activities

Stage Plan

Aspect Details
Owner Project Manager
Created For initiation stage: during SU; For subsequent stages: during SB at end of previous stage
Updated Not updated - if tolerance is forecast to be exceeded, create Exception Plan instead
Approved By Project Board
Purpose Detailed plan for the current management stage showing all products to be delivered and quality activities
Level of Detail Tactical - work package level with activity dependencies and resource assignments

Team Plan (Optional)

Aspect Details
Owner Team Manager
Created During Managing Product Delivery (MP) after receiving Work Package from Project Manager
Updated As needed by Team Manager to track detailed team activities
Approved By Team Manager (not formally approved by Project Manager)
Purpose Optional detailed plan for how the team will deliver assigned work packages
Level of Detail Operational - individual tasks and daily activities

Exception Plan

Aspect Details
Owner Project Manager
Created When stage or project tolerance is forecast to be exceeded (during CS or SB)
Approved By Project Board (stage exception) or Corporate/Programme Management (project exception)
Purpose Replaces the current Stage Plan (or Project Plan) when an exception is approved
Trigger Exception Report forecasting tolerance breach

Integration with Other Practices

Practice Integration Point
Business Case Plan costs and timescales feed Business Case; benefit realization dates influence planning
Quality Product Descriptions created during planning define quality criteria; quality activities scheduled in plans
Risk Risk responses are activities in plans; risk analysis may adjust estimates and schedules
Issues Off-specifications may require plan adjustments; change requests impact scope and schedule
Progress Plans provide baseline for progress measurement; actuals vs. plan drive exception management
