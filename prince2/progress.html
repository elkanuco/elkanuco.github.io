Progress Practice

The Progress practice provides ongoing monitoring and control to ensure the project remains viable and on track. It uses tolerance-based control, stage boundaries, and regular reporting to enable management by exception.

Progress Monitoring Activities

Setting Tolerances

Aspect Details
When During SU (project level) and SB (stage level)
Who Corporate/Programme sets project tolerance; Project Board sets stage tolerance; Project Manager sets work package tolerance
Six Tolerance Areas • Time: Acceptable deviation in schedule (e.g., ±2 weeks)
• Cost: Budget variance allowed (e.g., ±5%)
• Quality: Number/severity of quality criteria failures
• Scope: Products that can be removed/added
• Benefits: Acceptable reduction in benefits
• Risk: Overall risk exposure threshold
Purpose Define decision boundaries; enable management by exception; clarify escalation triggers

Tracking Progress

Aspect Details
When Continuously during CS and MP
Who Team Managers track work packages; Project Manager tracks stage; Project Board monitors project
Activities • Monitor work package completion
• Track actual vs. planned costs
• Compare actual vs. planned schedule
• Review quality results
• Monitor risk and issue status
• Forecast remaining work
• Identify variances and trends
• Update project records
Tracking Methods Checkpoint meetings, timesheet reviews, Earned Value Management, milestone tracking, product completion tracking

Reporting Progress

Report Type From → To Frequency Purpose
Checkpoint Report Team Manager → Project Manager Regular intervals (weekly/bi-weekly) Status of work packages, issues, risks, forecasts
Highlight Report Project Manager → Project Board Regular intervals (monthly typical) Stage status, achievements, issues, forecasts, next period plans
End Stage Report Project Manager → Project Board End of each management stage Stage performance review, lessons learned, next stage readiness
End Project Report Project Manager → Project Board Project closure Overall performance, benefits status, lessons learned, handover
Exception Report Project Manager → Project Board When tolerance forecast to be breached Exception details, options, impact, recommendation

Earned Value Management (EVM)

EVM Fundamentals

Metric Description Formula
Planned Value (PV) Budgeted cost of work scheduled to be completed by a specific date Baseline budget for scheduled work
Earned Value (EV) Budgeted cost of work actually completed % Complete × Budget
Actual Cost (AC) Actual cost incurred for work completed Sum of actual expenditures
Schedule Variance (SV) Measures schedule performance SV = EV - PV
Cost Variance (CV) Measures cost performance CV = EV - AC
Schedule Performance Index (SPI) Schedule efficiency ratio SPI = EV / PV
Cost Performance Index (CPI) Cost efficiency ratio CPI = EV / AC

Interpreting EVM Results

Indicator Good Performance Poor Performance
SV (Schedule Variance) Positive (ahead of schedule) Negative (behind schedule)
CV (Cost Variance) Positive (under budget) Negative (over budget)
SPI > 1.0 (work progressing faster than planned) < 1.0 (work progressing slower than planned)
CPI > 1.0 (getting more value per dollar) < 1.0 (spending more than budgeted)

Exception Management

Exception Process

Step Activity Details
1. Identify Exception Detect tolerance breach forecast Project Manager identifies that stage or project tolerance will be exceeded
2. Create Exception Report Document exception details Describe cause, impact, options, recommendation; submit to Project Board
3. Board Reviews Assess situation and options Project Board evaluates exception; may request additional analysis
4. Create Exception Plan Develop recovery plan Project Manager prepares Exception Plan replacing current Stage Plan (or Project Plan)
5. Board Approval Approve exception plan Project Board approves Exception Plan; project continues with new baseline

When to Raise an Exception

Trigger Description
Time Tolerance Breach Forecast completion date exceeds allowed tolerance
Cost Tolerance Breach Forecast costs exceed budget tolerance
Quality Breach Cannot achieve minimum quality criteria within tolerance
Scope Change Required scope change exceeds defined tolerance
Benefits at Risk Expected benefits will fall below tolerance
Risk Exposure Aggregate risk exposure exceeds risk tolerance
Business Case Invalid Business justification no longer sound

Stage and Event-Driven Control

Management Stage Boundaries

Aspect Details
Purpose Provide natural decision points for Project Board approval and direction
Activities at Stage End • Review stage performance (End Stage Report)
• Update Project Plan with actuals and revised forecasts
• Review and update Business Case
• Plan next stage (Stage Plan)
• Assess lessons learned
• Obtain approval to proceed
Typical Stage Boundaries Natural phases (design → build → test), decision gates, funding releases, risk mitigation points
Board Decision Continue to next stage, premature close, or request changes before proceeding

Event-Driven Controls

Event Control Response
Exception Forecast Immediate escalation; Exception Report to Project Board
Major Risk Materialization Ad-hoc report to Project Board; may trigger exception
Significant Issue Escalate to appropriate authority per Change Management Approach
Quality Failure Assess impact; may trigger off-specification or exception
External Change Review Business Case; update plans; inform Board if significant

Related Management Products

Highlight Report

Aspect Details
Owner Project Manager
Created Regularly during Controlling a Stage (CS) per agreed schedule
Recipient Project Board (may be distributed more widely)
Contains • Period covered
• Budget/schedule status
• Products completed this period
• Quality activities completed
• Key risks and issues
• Forecast vs. plan
• Plans for next period
• Decisions/actions needed

Exception Report

Aspect Details
Owner Project Manager
Created When stage or project tolerance is forecast to be exceeded
Recipient Project Board (stage exception) or Corporate/Programme (project exception)
Contains • Exception description and cause
• Consequences if no action taken
• Options available
• Impact of each option
• Recommendation
• Lessons learned

Integration with Other Practices

Practice Integration Point
Business Case Progress reports include Business Case status; poor performance may invalidate justification
Plans Progress measured against plans; variances inform replanning
Quality Quality results included in progress reports; quality failures may cause exceptions
Risk Risk status reported in progress reports; materialized risks impact progress
Issues Issue status reported; significant issues may trigger exceptions
